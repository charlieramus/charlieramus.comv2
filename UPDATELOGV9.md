# UPDATELOG V9 — ONE-FILE CONTENT: `site.config.ts` AS SINGLE SOURCE

**First read `DESIGN-BRIEF.md`, `AGENTS.md`, the V6 + V7 + V8 stage reports, and `MANUAL-TODO.md`.**
V6 made the site visually done; V7 made it a static `output: export`; V8 pulled curation into
`data/previews.ts`. **V9 is a pure content-architecture refactor** — it changes *where* editable
text and data live, not *what* renders. Today the editable surface is scattered across **10
hand-edited `data/*.ts` files** plus **~20 strings hardcoded inline in JSX** (section headings,
sublines, page ledes, nav labels — every `// CUSTOMIZE` marker in `components/` and `app/**/page.tsx`).
To tweak one line of copy you have to know which file — and sometimes which component — owns it.

**V9 collapses all of that into one file: `site.config.ts` at the repo root** — the single, heavily
commented, JSX-free surface a person edits to make this their site. This is the groundwork for
charlieramus.com becoming a **fork-and-customize open-source portfolio template**: edit one doc, run
`npm run build`, it's yours.

**The core noun this log introduces:**
**`site.config.ts` = the one editable file. All prose + hand-edited data. No JSX, no `className`, no imports to chase.**

This log **only relocates** content into one source and wires everything to read from it. It does
**not** change any rendered text, layout, style, route, or the V7 export pipeline — the site must
look **visually identical** before and after. It does **not** absorb the two things that are
generated or structural, not "copy you tweak": `data/photos.ts` (auto-generated by
`scripts/sync-gallery.mjs` from `public/photos/gallery.json`) and `data/motifs.ts` (the SVG-render
code registry). Those keep their own edit paths.

## Decisions (agreed in the CEO-review brainstorm)
- **Shape → typed single file, direct re-export (Option A).** `site.config.ts` holds the literal
  values; each old `data/*.ts` becomes a **thin re-export** of the same names/types from the config.
  Chosen over a YAML + `npm run` codegen because there's **one source of truth, no command to run,
  no generated files to drift**, and TypeScript catches a newcomer's typo the moment they make it.
- **Location → repo root `site.config.ts`** (imported `@/site.config`; the `@/*` → `./*` alias is
  confirmed). Root = maximally discoverable for a forker: "edit `site.config.ts`."
- **In scope (folds into the config):** `data/site.ts`, `about.ts`, `experience.ts`, `gear.ts`,
  `services.ts`, `socials.ts`, `writing.ts`, `projects-design.ts`, `projects-web.ts`, `previews.ts`
  — **plus** the inline JSX strings across `hero.tsx`, `personal-bento.tsx`, `about.tsx`, `work.tsx`,
  `right-now.tsx`, `petal-simon.tsx`, and `app/{design,gear,photography,web-projects}/page.tsx`.
- **Out of scope (stays put, referenced not moved):** `data/photos.ts` (sync-gallery output),
  `data/motifs.ts` (code registry). The config may *import their types* if it needs them; it never
  copies their data.
- **`// CUSTOMIZE` markers travel with the copy** — they now live in `site.config.ts` next to the
  value, so the annotation and the thing it annotates are finally in the same place.
- **Zero value changes.** Every string, number, order, and object shape is moved **character-for-
  character**. The proof is a before/after HTML export diff that comes back empty (§ Stage 4).
- **Medium refactor: four stages.** Two do the move (data files, then inline copy), one does the
  open-source docs, one is the visual sign-off gate.

## Hard constraints (carry over from V6 + V7 + V8)
- **Visually identical is the whole job.** No rendered text, layout, spacing, color, motion, or route
  may change. The gate is a **before/after diff of the exported HTML** (`out/`) across every route,
  with hashed `/_next/static/*` asset URLs normalized — it must be empty. If the diff is non-empty,
  the refactor changed something and the stage is not done.
- **Don't regress V6/V7/V8.** Keep **axe at 0**, no horizontal scroll, no console errors,
  reduced-motion honored, and **every route still prerenders under `output: export`**. The V7
  Cloudflare pipeline (`public/_redirects`, `public/_headers`) is untouched.
- **No new dependencies, no build step, no generated files.** The re-export shims are the only new
  code; they add no runtime cost (types erase, values are the same references).
- **Types stay resolvable.** Every downstream `import type { … } from "@/data/…"` and
  `import { … } from "@/data/…"` must keep working unchanged — the re-export files preserve the exact
  same export names and types, so the ~30 importer sites don't move.
- **Tailwind v4 is CSS-first** — this refactor touches **no CSS**. If a stage edits `globals.css`,
  that's a red flag; stop and reassess.
- Verify each stage with `tsc --noEmit` + `eslint .` + `next build` (export), then run the HTML diff.
  Don't commit/push unless Charlie asks.

## Stage 1 — `site.config.ts` scaffold + relocate the 10 data files

Create the single source and move the pure data into it, leaving thin re-exports behind. **No inline
JSX copy yet** (that's Stage 2) — this stage is the big, mechanical, provably-inert move.

- **New `site.config.ts` at the repo root** — one exported object (or a set of clearly grouped named
  exports) holding the *values* currently spread across the 10 hand-edited data modules:
  - `site.ts` → `SITE_URL`, `SITE_NAME`
  - `about.ts` → `snapshot`, `aboutParagraphs`, `tagline`, `finaleQuote`
  - `experience.ts` → the career timeline entries
  - `gear.ts` → the `gear` record
  - `services.ts` → `servicesHeading`, `servicesSub`, `services`
  - `socials.ts` → `socials`
  - `writing.ts` → the writing manifest entries
  - `projects-design.ts` → the design projects
  - `projects-web.ts` → the web projects
  - `previews.ts` → the curation config **and** its resolver functions (keep the resolvers with the
    config, or leave the resolvers in `data/previews.ts` re-exporting the moved config — either way
    the render sites' imports don't change)
- **Types:** move the shared type definitions (`Snapshot`, `Experience`, `GearItem`, `Social`,
  `WritingEntry`, `DesignProject`, `WebProject`, the preview pick types…) into `site.config.ts` (or a
  small `site.config.types.ts` the config imports). If the config needs a type from a file that
  **stays** (`Photo` from `photos.ts`, motif types from `motifs.ts`), **import it from there** — don't
  copy it. Keep the dependency graph acyclic: `site.config.ts` → (photos/motifs types only);
  `data/*.ts` → `site.config.ts`.
- **Convert each of the 10 `data/*.ts` files to a thin re-export** that preserves the exact same
  public surface, e.g. `export type { GearItem } from "@/site.config"; export { gear } from "@/site.config";`
  — same names, same types, same literal values. The ~30 downstream importers (`@/data/gear`, etc.)
  keep working with zero edits.
- **Leave `data/photos.ts` and `data/motifs.ts` untouched.** Confirm nothing in the move copies their
  data into the config.
- **Guardrail — zero value changes.** Before starting, build the export on the current commit and
  save it (`out.before/`). After the move, build again (`out.after/`) and diff the two HTML trees with
  `/_next/static/*` hashes normalized — **the diff must be empty.** That empty diff is the proof the
  relocation is inert.

  `data/*.ts` files are thin re-exports, the components read the config, and the `// CUSTOMIZE` notes
